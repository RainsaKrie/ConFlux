# Flux 继续开发交接说明

本文件的目标只有一个：让下一次会话中的开发者或 AI 助手只需先读这份文档，就能恢复当前项目状态，并沿着正确方向继续推进。

## 1. 推荐阅读顺序

1. [`06-HANDOFF.md`](./06-HANDOFF.md)
2. [`04-CHANGELOG.md`](./04-CHANGELOG.md)
3. [`02-ARCHITECTURE.md`](./02-ARCHITECTURE.md)
4. 必要时再读 [`01-PRD.md`](./01-PRD.md) 和 [`03-DATABASE.md`](./03-DATABASE.md)

如果时间非常有限，至少先读完本文件，再开始改代码。

## 2. 版本校正结论

这次交接最重要的上下文不是某个功能点，而是版本理解方式已经被重新校正：

- `v0.1 ~ v0.7` 可以视作相对线性的早期里程碑。
- `v0.8`、`v0.9`、`v1.0` 不是三次严格切开的独立发布，而是对当前主链能力的回顾性命名。
- 当前代码基线已经同时包含 `v0.9` 的段落级本地推荐主链和 `v1.0` 的长文切块主链。
- 同时，仍残留一部分本应属于 `v0.8` 的文案、术语与体验收口债务。
- 因此当前最准确的说法是：`v1.0 稳定化阶段`。

额外注意：

- `package.json` 的 `0.0.0` 不代表产品版本。
- 产品版本真相只以 [`04-CHANGELOG.md`](./04-CHANGELOG.md) 为准。

## 3. 当前代码里已经成立的事实

- `/graph` 的工程化拆分已经完成，核心逻辑位于 `src/features/graph/`。
- `Quick Capture` 已具备长文 `Semantic Auto-Chunking & Threading` 能力。
- 长文切块会生成共享 `threadId`，并通过 `project / source` 线程标签把同批碎片串联起来。
- 长文切片通过 `addBlocks()` 批量落盘，并默认跳过逐块 AI 打标。
- 编辑器中的旧式 `Phantom Weaving` 字级虚线、Hover Card 和整篇扫描面板已经下线。
- 当前 `Phantom Weaving` 只在当前段落上运行：`2500ms debounce + Entity Lexicon + Fuse.js` 本地预筛。
- 命中后只在右下角浮出“发现相关节点”的微型提示；点击后打开右侧 `Peek Drawer`。
- Drawer 顶部可执行：
  - `提取核心摘要`
  - `更新原文`
- 只有点击“更新原文”按钮时才真正调用 LLM，并继续复用现有 diff preview、revision history 与来源追溯链路。
- `/write` 的手动元数据分配已经进入可用态：主区完整暴露 `domain / format / project`，并以低存在感次级元数据条补充 `stage / source`。
- `/feed` 已可把 `stage / source` 纳入筛选建议与 Token 过滤；`/graph` 已可低存在感展示阶段与来源元数据。
- `Crystal Assimilation` 已具备“原文更新预览”、确认应用、线性版本历史、任意版本恢复与来源追溯。
- `activePoolContext` 与 `recentPoolEvents` 已经让 Feed、Write、Graph 具备一定连续体验。
- 开源版默认带有 onboarding seed 数据；当本地 `fluxBlocks` 为空时，store 会自动注入首批引导笔记。

## 4. 当前还没有完成的事

- 没有后端、同步、权限和协作能力。
- `stage / source` 已进入数据模型，并已在 `/write`、`/feed`、`/graph` 完成第一轮低存在感接入；但仍缺少批量管理、统计和规则化自动处理能力。
- 同化动作仍然是前端直连模型，没有缓存、任务队列、后端索引和协作能力。
- 站内仍有零星旧文案、旧术语和局部视觉语气未完全收束。
- docs 虽已完成本轮版本校正，但后续每次改动后仍需同步更新，否则很容易再次漂移。

## 5. 默认续开发协议

如果下次用户只说“继续”“看 docs 继续”“同步 docs 并推进”，默认按下面流程执行：

1. 先读取 [`06-HANDOFF.md`](./06-HANDOFF.md)、[`04-CHANGELOG.md`](./04-CHANGELOG.md)、[`02-ARCHITECTURE.md`](./02-ARCHITECTURE.md)。
2. 先确认当前工作仍应视为 `v1.0 稳定化阶段`。
3. 先核对文档与代码状态是否再次漂移；若漂移，优先同步 docs。
4. 默认优先做当前版本收口与下一阶段准备，而不是重新打开已经收口的大功能。
5. 若进入新阶段开发，优先补验证、边界、稳定性和可交接性。
6. 完成后默认运行与改动相关的校验：
   - 至少 `npm run build`
   - 涉及代码质量或页面逻辑时再加跑 `npm run lint`
7. 若本轮做了真实能力变更，默认同步更新：
   - [`04-CHANGELOG.md`](./04-CHANGELOG.md)
   - [`02-ARCHITECTURE.md`](./02-ARCHITECTURE.md)
   - 必要时更新 [`01-PRD.md`](./01-PRD.md)

## 6. 当前不应优先做的事

- 不要把 `/graph` 重新做成重型分析面板。
- 不要引入第二套关系模型或第二套图谱缓存。
- 不要因为 revision 已稳定，就立刻扩成复杂分支版本控制系统。
- 不要重新把编辑器变回“边写边骚扰”的字级提示工具。
- 不要在文档未同步的情况下持续推进多个新方向。
- 不要继续把 `v0.8`、`v0.9`、`v1.0` 写成三次干净独立发布。

## 7. v1.0 稳定化阶段的默认优先级

如果没有新的更高优先级指令，下一阶段默认优先做这些事：

1. 继续清理残留旧文案、乱码、命名漂移与样例数据中的历史语义。
2. 继续减轻遗留样例文件和重复状态逻辑带来的维护成本。
3. 继续减轻页面层的小型技术债：
   - 重复状态更新
   - 临时调试遗留
   - 轻量组件内的耦合逻辑
4. 继续强化段落级推荐的命中质量、去重策略和同化失败恢复。
5. 为长文切块补更多固定样本与回归验证。

## 8. 推荐验证命令

继续开发前后，至少执行：

```bash
npm run build
```

如果本次涉及页面交互、状态流或组件重构，再执行：

```bash
npm run lint
```

如果本次涉及段落推荐、长文切块或推荐词典调整，再补跑：

```bash
npm run verify:phantom
```

## 9. 交接结论

下次如果用户只说“读 docs 后继续”，默认理解为：

- 先同步 `docs`
- 先检查当前项目是否存在明显技术债
- 优先做 `v1.0` 稳定化收口，而不是继续跳版本
- 保持验证通过
- 再把最新状态回写文档
