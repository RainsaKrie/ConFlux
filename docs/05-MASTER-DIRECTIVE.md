# Flux MVP 终极演进架构

副标题：零摩擦输入与双向知识同化

## 1. 为什么需要这份指令

Flux 已经完成了第一阶段 MVP：

- 我们有扁平的 `fluxBlocks`
- 我们有 `/feed`、`/write`、`/graph` 三个视角
- 我们有 Quick Capture、标签系统、Adaptive Lens、Peek Drawer 和 Command Deck

但这些能力还主要停留在“用户主动操作，系统被动响应”的阶段。

下一阶段的核心问题是：

- 用户不该总被要求先知道要 `@` 谁
- 系统不该只是把旧知识搬过来，而应该能够让新知识长回旧知识
- UI 不该为了“智能感”变得压迫、厚重或打断心流

这份文档定义 Flux 的下一阶段北极星：`主动预期`、`零压迫 UI`、`双向生长`。

## 1.1 当前落地状态

截至当前 `v0.5.1`，这份指令已经不再只是愿景，已有一部分进入实现态：

- `Phantom Weaving` 已具备标题级本地静默嗅探、虚线 Decoration、Hover Card 与抽屉联动。
- `Crystal Assimilation` 已具备最小可用闭环，可把当前新段落静默回流到母体。
- `Adaptive Lens` 已从厚重卡片脱水为更接近正文的引述流。
- `Peek Drawer` 已成为真正可并列参考的右侧阅读面板。
- `/graph` 已完成 `Semantic Zoom + Spotlight Search + Cluster Framing`，并且相关实现已经完成工程化拆分。

也就是说，Flux 已经从“被动引用工具”迈入“主动预期 + 双向生长”的早期阶段。

## 2. 北极星原则

### 2.1 主动预期

系统要在后台静默理解用户当前正在写什么，并主动发现可能相关的知识块，而不是永远等待用户显式输入命令。

### 2.2 零压迫 UI

界面必须继续遵守 `Zen Canvas`：浅灰背景、白色卡片、大留白、低噪声交互。智能提示应该像呼吸，而不是像警报。

### 2.3 双向生长

知识引用不应只有单向“摘取”。真正的个人外脑，需要支持“当前新洞察反向同化回旧知识母体”。

## 3. 三大演进模块

### 模块一：Zen UI 与 Gravity Pools

目标：让界面大方、模块化、一目了然。

原则：

- 废弃“文件夹思维”，保留 `Gravity Pools` 作为筛选视角快照。
- 强化顶部流式输入，不要求先填标题再保存。
- 元数据默认低存在感，只在需要时浮现。
- Feed、Editor、Graph 三页都要保持同一套浅色克制视觉秩序。

产品含义：

- 用户面对的是知识流，不是目录树。
- 用户保存的是观察方式，而不是容器所有权。

### 模块二：Phantom Weaving

目标：在不打断心流的前提下，让系统静默发现潜在知识关联。

交互要求：

- 监听 TipTap 输入，但不要在高频打字时抢事务；应采用深度防抖与 idle 时机。
- 优先使用本地检索，不允许每次输入都调用模型。
- 若发现高置信候选，在正文中对相关词语绘制极轻量虚线 Decoration。
- 用户可以完全忽略，也可以 Hover 查看详情。

技术要求：

- 第一阶段应优先考虑标题级嗅探，宁可漏报也不要误报。
- 需要统一复用 `contentToPlainText()`，保证 `Adaptive Lens` 摘要不会在检索时丢失。
- Decoration 必须足够克制，避免形成高压提醒感。
- 必须内建 Anti-Overlinking 机制：短词过滤、停用词过滤、严格阈值、单母体单次命中、默认几乎隐形。

### 模块三：Crystal Assimilation

目标：让新写出的增量洞察，能优雅地回流并更新旧知识母体。

交互要求：

- Hover 幽灵虚线时弹出轻量浮层。
- 浮层至少提供两个动作：`提取摘要块` 与 `逆向同化`。
- `提取摘要块` 保留当前 Adaptive Lens 路径。
- `逆向同化` 则触发真正的知识回流闭环。

业务逻辑：

1. 截取当前新段落 `NewContent`
2. 读取目标母体全文 `TargetBlock`
3. 调用模型生成“结构化融合后的母体正文”
4. 静默写回目标 block
5. 在当前正文旁留下低压迫的成功挂件，提示这段新知识已被收录

设计约束：

- 不允许粗暴覆盖用户当前输入。
- 不允许简单把内容 append 到文末。
- 必须强调“融合进合适上下文”，必要时由模型新建小标题。

## 4. 推荐实施顺序

### v0.3 回顾：Phantom Weaving 原型

原因：

- 这是所有“主动预期”能力的入口。
- 只涉及本地检索、Decoration 与轻量 Hover UI，风险最可控。
- 做通这层后，才能自然挂接 `提取摘要块` 与 `逆向同化`。

验收标准：

- 用户在编辑器中连续输入时，不会感到明显卡顿。
- 高置信命中的词语出现轻量虚线。
- Hover 后能看到候选母体标题和两个动作入口。
- 默认场景下不会出现满屏虚线或重复命中同一母体的噪音。

### v0.4 回顾：Crystal Assimilation

原因：

- 这是最难的业务闭环，涉及 AI 融合改写与数据回写。
- 它必须建立在稳定的“候选发现机制”之上。

验收标准：

- 模型能返回结构上合理的母体融合文本。
- 回写母体后，Feed、Write、Graph 都能反映出最新内容。
- 当前写作流不被打断。
- 用户在当前正文中能收到低压迫但明确的“已收录”反馈。

当前已完成的最小闭环：

- Hover 卡可直接触发 `逆向同化`
- 成功后静默写回目标母体
- 正文附近会浮出低压迫挂件，可一键打开右侧抽屉核对结果

### v0.5 回顾：零压迫知识流

当前收口目标：

- Feed 能理解回流后的摘要变化。
- Editor 能展示“已同化”的来源挂件和抽屉入口。
- Graph 能更自然地呈现知识密度与关系演化。

当前结论：

- `v0.5` 已经完成。
- `Gravity Pool` 已成为三视图共享的“观察语境”。
- `Phantom Weaving` 和 `Crystal Assimilation` 已不再只发生在编辑器局部，而是会回流到当前 Feed / Graph 主题。
- 当前剩余的主要问题正式进入 `v0.6` 范畴：实体字典、版本历史与引用语言统一。

## 6. 当前结论

Flux 的下一阶段，不是继续追加零散功能，而是完成一次产品范式转换：

- 从被动笔记，转向主动预期
- 从单向引用，转向双向生长
- 从知识展示器，转向会自我整理的个人外脑

## 7. 下一步建议

如果继续沿着当前实现推进，我建议遵循下面这个顺序，而不是再平铺新功能：

1. `v0.6` 的 `Entity Lexicon` 升级已经完成，让实体词条而不只是完整标题参与嗅探。
2. `v0.6` 接下来做 `Crystal Assimilation` 的 Diff Visualization、持久历史、抽屉核对整合与质量控制。
3. `v0.7` 再做 `Phantom Weaving` 的固定测试样本与命中质量调优。
4. `v0.8` 统一 `Adaptive Lens`、Hover Card、Peek Drawer、成功挂件的脚注式引用语言。
5. 在 `v0.8` 之后，才考虑把嗅探从标题级扩到标签、透镜摘要和更轻量的正文短语召回。

原因很简单：

- 当前最大的风险已经不是“没有入口”，而是“入口有了以后，质量是否可控、是否可追踪”。
- 如果先扩召回范围而不补质量控制，很容易再次滑向过度关联和视觉噪音。
